A1 - Injection

  • Injection av kode i forskjellige kontekster
  • SQL, LDAP, XML
  • OS kommandoer, XPath
  • Vi må alltid holde data adskilt fra kommandotegn
  • Fortsatt på toppen, mye grunnet sine mange angrepsvinkler
  • DEMO
sql = "SELECT name FROM users WHERE name like '%#{params[:q]}%'"
  • http://www.itsecurityguru.org/2016/12/06/dailymotion-loses-85-million-users-details-data-breach-industry-reaction/

A2 - Broken Authentication and Session Management

  • Passord i klartekst, både ved lagring og overføring
  • Session fixation (jsessionid=random)
  • Lang eller manglende timeout på sessions
  • "/><script>alert(1);</script><foo id="

A3 - Cross-Site Scripting (XSS)

  • Stored XSS
  • Reflected XSS
  • DOM Based XSS

  • Escape alt i riktig kontekst, håndteres vanligvis av respektive rammeverk
  • Ta i bruk CSP for et ekstra nivå av sikkerhetsnett
  • X-XSS-Protection - Vanligvis default satt, hindrer bl.a. noe reflected XSS
  • https://www.digi.no/artikler/16-mai-ble-det-full-kriseberedskap-i-evry-etter-datalekkasje-rotarsaken-kan-pavirke-alle-norske-selskaper/383149

A4 - Broken Access Control

  • Samlekategori med deler fra A7 og A4 fra 2013
  • Manglende autorisering
  • F.eks. administrasjonsgrensesnitt
  • Fuzzy sikkerhet
  • Hvis en angriper vet referansen til objektet, så kan han besøke siden uten autentisering
  • Skjedde nylig med fakturahotell og EVRY som ble indeksert av Bing
  • Bruk indirekte referanser i stedet for direkte referanser hvis mulig
  • DEMO
  • https://www.helpnetsecurity.com/2016/06/23/154-million-us-voter-records-exposed/

A5 - Security Misconfiguration

  • Default passord
  • Feilhåndtering med stacktrace
  • Rammeverk med automatisk admin grensesnitt
  • Applikasjon i debug mode
  • Feilkonfigurert lastbalanserer/webserver etc.
  • Database åpen mot nettet, og gjerne uten passord
  • DEMO
  • http://gulfnews.com/xpress/news/data-of-34-million-keralites-leaked-in-massive-breach-1.1930317

A6 - Sensitive Data Exposure

  • Bruker ikke TLS (for sensitive data)
  • Data blir eksponert ukrypert
  • Sensitive data blir lagret uten/utdatert kryptering
  • Api som er for verbost
  • Mangler headere f.eks. secure flag på cookie

A7 - Insufficient Attack Protection

  • Ny kategori av året
  • Diskutabel pga. knytning mot kommersielle aktører
  • Men høyst reell, mange applikasjoner blir angrepet daglig av automatiske (blinde) angrep
  • En av de første til å ta med devops i listen
  • Bør kunne detektere unormal trafikk, f.eks. automatisk sperre bruker/ip etter gjentatte påloggingsforsøk/sql exceptions etc

A8 - Cross-Site Request Forgery (CSRF)

  • Manglende sjekk av at brukeren faktisk gjorde requestet
  • Sjekkes ved å ha f.eks. et unikt token som er generert til brukeren
  • Kan ikke alltid stole på innebygd støtte, f.eks. Rails sjekker ikke csrf token i get request
  • http://www.wired.co.uk/article/panama-papers-mossack-fonseca-website-security-problems

A9 - Using Known Vulnerable Components

  • Bruker sårbare plugins eller rammeverk
  • Kan minske risikoen ved å bruke en dependency check
  • Et relativt stort problem for mange nettsider som "bare kjører"
  • Ta i bruk OWASP Dependency Check, retire.js, Brakeman etc.
  • Eks. Mossack Fonseca (Panama papers) skjedde pga. kjent sårbarhet i Drupal
  • http://www.datasikkerhetsboka.no/blogg/2017/02/02/ae-ha-itj-gjort-no-galt/

A10 - Underprotected APIs

  • Veldig utbredt med enkle api som f.eks. server mobilapper,iot devices etc.
  • Disse får ikke alltid samme fokus på sikkerhet som tradisjonelle web apps
  • Lett å overse mange av de andre punktene ettersom man ikke er "koblet til web"
  • Kan fort ende opp med passord lagret i appen, manglende escape av input som fører til sql injection eller andre ting som er nevnt i ovenstående punkter
  • Rema 1000 "Æ"
